# 2026-07-12 개발 로그

## 오늘 한 것 요약
- **커뮤니티 목록 쿼리 성능 최적화**: `findPostPage`(+ 관리자·내글 목록)의
  "전체 조인·집계 후 페이지네이션" 안티패턴을 "페이지네이션 먼저" 구조로 재설계.
- **커버링 인덱스 추가**: `COMMUNITY_POST(is_blinded, created_at)`, `(user_id, created_at)`.
- 100만 게시글·200만 댓글 더미로 재현 측정: **14,046ms → 0.79ms** (약 1.8만 배).

---

## 배경 — 실제로 병목이 났나?
아니다. 가당은 실사용 게시글이 소규모라 실운영 병목은 없었다.
다만 "DB 성능 최적화"를 말로만 아는 게 아니라 실제로 할 줄 아는지 확인하려고,
게시판 목록 쿼리가 대규모에서 어떻게 무너지는지 **더미 100만/200만 건으로 재현**해서 개선했다.

## 문제 — 왜 14초였나
구 `findPostPage`:
```sql
SELECT p.*, COUNT(c.comment_id) AS comment_count
FROM COMMUNITY_POST p
JOIN USER u ...
LEFT JOIN COMMENT c ON c.post_id = p.post_id   -- 200만 댓글 전부 조인
WHERE p.is_blinded = 0
GROUP BY p.post_id, ...(14개 컬럼)              -- 100만 그룹 집계
ORDER BY p.created_at DESC                       -- 인덱스 없어 filesort
LIMIT 20;                                        -- 마지막에 20개만
```
DB 실행 순서상 **20개를 고르려고 100만 건을 조인·집계·정렬**한다.
`LIMIT 20`은 앞 단계가 이미 전량을 건드린 뒤라 일을 못 줄인다 (aggregate-then-paginate 안티패턴).

## 왜 인덱스만으론 안 됐나 (14s → 21s, 오히려 악화)
`(is_blinded, created_at)` 인덱스만 추가하면 정렬은 공짜가 되지만
**여전히 200만 댓글을 다 조인·집계**해야 한다. 옵티마이저가 인덱스 순서로 접근하려다
랜덤 I/O가 늘어 오히려 느려졌다. **교훈: 인덱스는 잘못된 쿼리 구조를 못 고친다.**

## 해결 — 페이지네이션을 맨 앞으로 (구조 재설계)
```sql
SELECT p.*, u.nickname,
       (SELECT COUNT(*) FROM COMMENT c WHERE c.post_id = p.post_id) AS comment_count
FROM (
    SELECT post_id                     -- ① post_id만 얇게
    FROM COMMUNITY_POST
    WHERE is_blinded = 0
    ORDER BY created_at DESC
    LIMIT 20 OFFSET 0                   -- ② 인덱스로 20개 먼저 확정
) ids
JOIN COMMUNITY_POST p ON p.post_id = ids.post_id   -- ③ 그 20개만 상세 조인
JOIN USER u ON u.user_id = p.user_id
ORDER BY p.created_at DESC;
```
- 안쪽 서브쿼리는 `(is_blinded, created_at)`가 **커버링 인덱스**로 작동 →
  정렬이 인덱스에 이미 돼 있어 역방향으로 20칸 읽고 멈춤(테이블 본체 접근 0).
- 조인·집계·댓글 COUNT를 **확정된 20건에만** 수행 → 각 20회.
- 건드리는 행 수: 100만+200만 → 약 20+α. **본질은 "읽는 행 수를 줄인 것".**

## 딥페이지 — OFFSET vs 키셋
재설계해도 `LIMIT 20 OFFSET 500000`은 110ms (OFFSET은 앞 50만개를 세서 버림).
**키셋(커서) 페이지네이션**으로 해결 가능:
```sql
WHERE is_blinded=0 AND created_at < :last_seen   -- OFFSET 대신 커서
ORDER BY created_at DESC LIMIT 20                  -- 인덱스가 커서로 점프 → 20칸만
```
→ **1.08ms, 페이지 깊이 무관** (무한 스크롤에 적합). *현재 API는 offset 방식 유지,
키셋은 향후 무한스크롤 전환 시 도입 여지로 문서화만.*

## 실측 (100만 게시글 / 200만 댓글, EXPLAIN ANALYZE)
| 버전 | 1페이지(offset 0) | 딥페이지(offset 50만) |
|---|---|---|
| 원본 (인덱스 X) | **14,046 ms** | 더 나쁨 |
| + 인덱스만 | **21,001 ms** (악화) | — |
| 인덱스 + 구조 재설계 | **0.79 ms** | 110 ms |
| + 키셋 | — | **1.08 ms** |

## 면접 포인트
"인덱스 걸면 빨라진다"의 착각을 실측으로 깬 서사. **진짜 병목은 쿼리 구조**(집계-후-페이지네이션)였고,
페이지네이션을 앞세워 읽는 행 수 자체를 줄였다. 인덱스·커버링 인덱스·키셋 페이지네이션 3층을
숫자(14s→0.8ms)로 설명 가능. 단, **소규모라 선제적 개선**이었음을 정직하게 밝힐 것.
