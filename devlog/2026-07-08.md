# 2026-07-08 개발 로그

## 오늘 한 것 요약
- 인증 개선: access 토큰 단일(24h) → **access 30분 + refresh 14일(HttpOnly 쿠키 + Redis)**
- refresh rotation: 재발급마다 refresh 토큰 교체, Redis `getAndDelete`로 원자 소모
- 프론트 axios 401 인터셉터: 자동 재발급 + 원 요청 1회 재시도 (동시 401은 단일 refresh로 합침)

---

## 1. 왜 고쳤나 — 기존 구조의 문제

기존: access 토큰 하나(만료 24시간)를 localStorage에 저장.
- **XSS 한 번이면 24시간짜리 토큰이 통째로 유출**되고, JWT는 서버가 무효화할 수단이 없음
- 만료를 줄이면 사용자가 하루에도 몇 번씩 로그아웃되는 트레이드오프
- 로그아웃해도 토큰은 만료 전까지 계속 유효 (클라이언트에서 지울 뿐)

## 2. 설계 결정

| 결정 | 이유 |
|---|---|
| access 30분으로 단축 | 유출 시 피해 창을 24h → 30분으로 |
| refresh는 **불투명 난수**(JWT 아님) | 어차피 서버 저장이 전제 → 서명 불필요. Redis 삭제 = 즉시 무효화 |
| refresh 저장소 = **Redis** (`refresh:{token}`→userId, TTL 14d) | 이미 L2 캐시로 도입한 Redis 재활용. 만료는 EXPIRE에 위임 (L2 캐시와 동일 패턴) |
| refresh 전달 = **HttpOnly 쿠키** (`Path=/api/auth`, SameSite=Lax) | JS 접근 불가 → XSS로 refresh까지 탈취 불가. Path 제한으로 일반 API에는 안 실림 |
| **rotation**: 재발급마다 old 소모 + new 발급 | 탈취된 refresh가 재사용되면 이미 소모된 키라 실패 → 세션 강제 종료 |
| 소모는 `opsForValue().getAndDelete()` | 조회+삭제 원자 처리 — 동시 재발급 경합/재사용 공격이 두 번째부터 null |

access는 여전히 localStorage(메모리성 사용). refresh가 쿠키로 보호되므로
access 유출의 피해는 최대 30분으로 제한된다.

## 3. 변경 내역

**백엔드**
- `auth/RefreshTokenService` 신규: issue / consume(원자 소모) / revoke
- `auth/RefreshCookie` 신규: HttpOnly·SameSite=Lax·Path=/api/auth 쿠키 생성/만료
- `auth/AuthTokens` 신규: (body용 AuthResponse, 쿠키용 refreshToken) 쌍
- `AuthService.refresh()/logout()` 추가, 로그인·회원가입·OAuth 전부 토큰 쌍 발급으로 변경
- `AuthController`에 `POST /api/auth/refresh`, `POST /api/auth/logout` 추가 (SecurityConfig permitAll)
- `jwt.expiration` 86400000 → 1800000, `jwt.refresh-expiration-days=14` 신설

**프론트**
- axios `withCredentials: true` (쿠키 전송, CORS는 이미 allowCredentials)
- 401 응답 시 `/auth/refresh` 호출 → 새 access 저장 → 원 요청 재시도(`_retried` 1회 가드)
- 동시 다발 401은 `refreshPromise` 공유로 refresh 1회만 발사
- `useAuth.logout()`이 서버 revoke(`/auth/logout`)까지 호출

**테스트**: `AuthServiceTest`에 rotation·재사용 거부·로그아웃 revoke 테스트 4건 추가.

## 4. 삽질/함정 기록

- **JwtAuthenticationFilter가 만료 토큰이면 permitAll 경로도 401로 끊음** —
  프론트 인터셉터가 만료된 access를 `/auth/refresh`에도 자동 첨부해서 재발급 자체가 죽는 구조.
  → refresh/logout 호출에는 Authorization 헤더를 생략(쿠키만 필요)으로 해결.
  (근본 대안: 필터가 파싱 실패 시 401을 직접 쓰지 말고 인증 없이 체인을 계속 태워
  EntryPoint에 위임하는 것 — 표준 패턴이지만 전 엔드포인트 동작이 바뀌므로 마감 후 과제로)
- localhost:5173 → localhost:8080은 CORS상 cross-origin이라 `withCredentials` 필수,
  단 SameSite 관점에선 same-site(포트 무시)라 Lax여도 쿠키가 실린다.

## 5. 검증 상태

⚠️ 이 컴퓨터에 JDK·Docker가 없어 **컴파일/기동 검증 미완**. 집 컴에서:
1. `docker compose up -d --build` 후 로그인 → 응답 헤더에 `Set-Cookie: gadang_refresh=...` 확인
2. `jwt.expiration=10000`(10초)으로 낮추고 만료 후 아무 인증 API 호출 → 네트워크 탭에서
   401 → `/auth/refresh` 200 → 원 요청 재시도 200 순서 확인
3. refresh 직후 이전 쿠키 값으로 `/auth/refresh` 재호출 → 401 (rotation 확인)
4. 로그아웃 후 `/auth/refresh` → 401, Redis에서 `KEYS refresh:*` 비어 있는지 확인
5. `./mvnw test` — AuthServiceTest 7건
