## 1. 시크릿 분리 (보안)

**문제**: `application.properties`에 카카오·ODsay·네이버·TourAPI 실 키와 GMS 키가 평문 하드코딩 + git 추적됨. 루트 `.env`도 실 키째 추적됨.

**조치**
- `application.properties`의 모든 외부 키 → `${ENV:}` placeholder로 교체
- `spring.config.import=optional:classpath:application-local.properties` 추가
- 실제 키는 git 제외된 `application-local.properties`로 이동 (+ `.example` 템플릿)
- 루트 `.env` → `git rm --cached` + `.gitignore`에 `.env` 추가 (+ `.env.example`)
- DB 계정 기본값을 `gadang/gadang` 으로 변경

**남은 일**: 키들이 git 히스토리에 남아있음 → 정석은 키 재발급 + 히스토리 재작성(filter-repo/BFG). 포트폴리오용으로는 "이후 추적 안 됨 + 재발급 권장"으로 문서화.

## 2. 장소 조회 캐시 통일 — GPS·지도·지역명 → 같은 지역 캐시

**문제**: 같은 지역이라도 진입 경로에 따라 캐시가 갈렸음.
- 지역명 선택: `resolveRegionCenter(region)` → 지역 표준 중심좌표 → 격자 키
- GPS·지도 좌표: 원본 좌표 그대로 → 다른 격자 셀 → **다른 캐시 키 → 동일 지역인데 캐시 미공유 + 결과 불일치**

좌표→지역 변환(`coordToSiGun`)은 `filterByCoord` 내부에서만 일어나 **캐시 키엔 반영 안 됨**.

**조치**
- `PlaceFilterService.snapToRegion(lat, lng)` 추가:
  - `coordToSiGun(lat,lng)`로 지역명 판정 → `resolveRegionCenter(region)`으로 표준 중심좌표 변환
  - 역지오코딩 실패 시 `region=null`로 원본 좌표 유지(안전 fallback)
  - 반환: `RegionSnap(lat, lng, region)` record
- `PlaceSuggestController`의 좌표 경로 2곳(`/suggest`, `/suggest/stream`)이 모두 `snapToRegion` 경유

**효과**: GPS·지도·지역명이 같은 지역이면 동일한 표준 중심좌표 → **동일 격자 키 → 캐시 공유 + 결과 일관**.
`coordToSiGun`이 광역시는 "부산", 도는 "경주"처럼 반환 → REGION_META 키 형식과 일치 확인.

**결정**: 제품 방향을 "지역 단위 조회"로 확정. (지도에서 콕 집은 위치 기준이 아니라, 같은 지역이면 같은 결과)

**확인**: `./mvnw -o compile` → BUILD SUCCESS.

## 3. 코스 결과 화면 — 방문 순서 경로 지도

**목표**: 코스 생성 완료 결과 페이지에서 방문 순서대로 경로를 지도에 표시.

**프론트 변경 (`CourseView.vue`)**
- `routePlaces` computed: `result.items` 중 `type === 'place'` + 좌표 있는 항목 (배열 순서 = 방문 순서)
- `renderResultMap()`: 기존 `loadKakaoSDK()` 재사용
  - 방문 순서 번호 CustomOverlay 마커(1,2,3…)
  - 순서대로 잇는 `kakao.maps.Polyline` 경로선
  - `LatLngBounds`로 전체 경로 보이게 범위 맞춤
  - 재생성 시 이전 마커·선 정리
- `watch(result)`: 결과 생성/재생성 시 `nextTick` 후 지도 다시 그림. result=null이면 맵 핸들 초기화.
- 템플릿: 결과 header와 `CourseTimeline` 사이에 지도 카드 삽입 (`routePlaces.length > 0`일 때만)

**미확인**: 프론트 `node_modules` 미설치로 빌드 미실행. `pnpm install && pnpm dev`로 동작 확인 필요.

## 4. 캐시 stampede 방어 — @Cacheable(sync=true)

**문제**: `@Cacheable`에 sync가 없어, 콜드 상태에서 동일 키 동시 요청 N개가 전부 miss →
N번 외부 API 호출(cache stampede). ODsay 쿼터 소진 + Tomcat 스레드풀 고갈 위험.

**조치**: stampede에 가장 취약·고비용인 두 캐시에 `sync = true` 적용
- `RealScoredPlaceProvider.getScoredPlaces` (placeCandidates, 2개 오버로드)
- `NaverDataLabService.getAverageRatios` (datalabTrend)
→ 같은 키는 한 스레드만 계산, 나머지는 대기 후 결과 재사용. 1000 동시요청 → 외부 호출 1번.

**제외**: `NaverBlogService.getBlogCount`는 condition·unless가 있어 sync와 비호환. 나머지는 단일 호출이라 후순위.

**한계**: 단일 인스턴스 내 한정. 수평 확장 시엔 Redis 분산 락 필요(발전 과제).

**확인**: `./mvnw -o compile` → BUILD SUCCESS.
