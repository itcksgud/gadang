## 1. 서킷브레이커 (P5)

### 왜 — 타임아웃만으로 부족한 지점

RestClient에 connect 3s / read 10s 타임아웃은 이미 있었다. 하지만 ODsay가 죽어 있으면
**모든 요청이 각자 10초씩 스레드를 물고 실패**한다. 부하 상황이면 Tomcat 200스레드가
외부 대기로 소진 → 무관한 API까지 멈추는 연쇄 장애("트래픽 몰리면?" 답변의 P5 지점).
서킷브레이커는 실패가 누적되면 **기다리지 않고 즉시 실패**시켜 스레드를 지킨다.

### 설계 결정

| 결정 | 이유 |
|---|---|
| Spring Boot 어댑터(`resilience4j-spring-boot3`) 대신 **코어 직접 사용** | pom이 Spring Boot **4.0.6** — boot3 어댑터의 호환 미검증 리스크 회피. 코어는 프레임워크 무관 |
| 적용 지점 = **RestClient 인터셉터** (`CircuitBreakerInterceptor`) | 서비스 코드 무수정. 브레이커 OPEN → `CallNotPermittedException`(런타임) → 각 서비스의 기존 `catch → null → fallback`(Haversine·BusFrequencyTable)이 그대로 작동 |
| **API별 이름 붙은 브레이커 7개** (kakao/odsay/naver/tour/openMeteo/rag/korail) | ODsay 장애가 Kakao 호출까지 차단하면 안 됨 — 장애 격리 단위 = 의존성 단위 |
| **4xx는 성공으로 집계** (5xx·IO예외만 실패) | Naver 429는 우리가 요청을 줄일 문제(rate limit)지 상대 장애가 아님. 429로 브레이커가 열리면 멀쩡한 API를 스스로 끊는 오작동 |
| slow call 기준 5s / 80% | read timeout(10s)까지 가기 전에 "느려짐"을 선제 감지해 차단 |

파라미터: 최근 20건 슬라이딩 윈도우, 최소 10건 관측, 실패율 50%에 OPEN,
30초 후 HALF_OPEN에서 시험 호출 3건으로 복구 판정.

### 변경 파일
- `config/ResilienceConfig.java` 신규 — `CircuitBreakerRegistry` 빈 + 마이크로미터 바인딩
- `config/CircuitBreakerInterceptor.java` 신규 — `ClientHttpRequestInterceptor` 구현
- `config/RestClientConfig.java` — 7개 클라이언트에 `.requestInterceptor(breaker(...))` 부착
- 테스트 `CircuitBreakerInterceptorTest` 4건: IO 실패 누적→OPEN→즉시 차단(외부 미호출 검증),
  5xx 실패 집계, 4xx 비집계, 정상 통과

## 5. JWT 필터 표준 패턴 리팩터링 (7/8 devlog의 "마감 후 과제" 조기 처리)

**문제**: `JwtAuthenticationFilter`가 토큰 파싱 실패 시 401을 직접 쓰고 체인을 끊음 —
만료 access가 자동 첨부되면 permitAll인 `/api/auth/refresh`까지 죽어서 프론트가
refresh 호출에만 Authorization 헤더를 생략하는 우회를 갖고 있었다.

**해결**: 무효 토큰이면 SecurityContext만 비우고 미인증 상태로 체인 계속.
거부 판단은 SecurityConfig의 EntryPoint(보호 엔드포인트 401)에 위임 — 필터는
"인증 정보 싣기"만, "거부"는 인가 계층이 하는 표준 책임 분리.

**실기동 검증** (재빌드 후 실호출):
| 요청 | 결과 |
|---|---|
| permitAll(`/api/places/regions`) + 쓰레기 토큰 | **200** (기존엔 401로 죽던 케이스) |
| 보호(`/api/mypage/profile`) + 쓰레기 토큰 | 401 (EntryPoint, 동일 JSON 포맷) |
| 보호 + 토큰 없음 | 401 |

  ①에 가려 드러나지 않았을 뿐, ①만 고치면 이게 터지는 이중 구조.
  → ci.yml에 MySQL(health check 대기)·Redis service 컨테이너 추가.

교훈: "CI를 만들었다" ≠ "CI가 돈다". **초록불을 눈으로 확인하기 전까지는 없는 것.**

## 8. 삽질 기록 — Docker Desktop 크래시 루프 (이 PC 한정)

이 PC(Windows 11 빌드 26200 + Docker Desktop 4.81)에서는 **Docker가 만든 유닉스 소켓
파일이 프로세스 종료 후 접근 불가능하게 깨진다** (`fsutil`조차 오류 123). Docker가 기동
시 이전 소켓을 지우려다 실패 → "initializing Inference manager/Secrets Engine:
remove ...: The file cannot be accessed by the system" 크래시 무한 루프.

- 함정 1: 파일 삭제(`Remove-Item`)는 조용히 실패한다 — 깨진 소켓은 일반 삭제 불가.
- 함정 2: 크래시한 세션이 좀비 프로세스로 남아 `Stop-Process`도 안 먹고, 새로 실행하면
  기존 인스턴스에 합류만 됨 → `taskkill /F /T`로 트리째 죽여야 함.
- 함정 3: 깨진 소켓이 **두 군데**(`%LOCALAPPDATA%\Docker\run`, `%LOCALAPPDATA%\docker-secrets-engine`)라
  하나만 치우면 다른 쪽에서 죽고, 그 시도가 새 깨진 소켓을 남긴다 — **동시에 치워야** 탈출.
- 해법: 삭제 대신 **폴더 이름 변경**(rename은 깨진 파일을 안 건드림) 후 재시작:

```powershell
taskkill /F /IM "Docker Desktop.exe" /T; taskkill /F /IM "com.docker.backend.exe" /T
Rename-Item "$env:LOCALAPPDATA\Docker\run" "run_broken_$(Get-Date -Format HHmmss)"
Rename-Item "$env:LOCALAPPDATA\docker-secrets-engine" "secrets_broken_$(Get-Date -Format HHmmss)" -ErrorAction SilentlyContinue
```

## 6. 게시글 이미지 저장 — 로컬 디스크 → S3 호환 오브젝트 스토리지

**문제 2가지**
1. 앱 서버 로컬 디스크(`uploads/`) 저장 → 수평 확장 시 인스턴스마다 파일이 갈라지고,
   compose에 볼륨도 없어 컨테이너 재생성 시 전부 유실.
2. 확장자를 클라이언트 파일명에서, MIME을 클라이언트 Content-Type에서 그대로 신뢰 →
   `image/png`으로 위장한 `x.html` 업로드 시 `/files/uuid.html`로 저장·정적 서빙
   (permitAll GET) = **stored XSS**.

**해결**
- `ImageStorage` 인터페이스로 저장소 추상화. `gadang.storage.mode`로 선택:
  - `local`(기본): 기존 동작 유지 (개발·테스트)
  - `s3`: AWS SDK v2 — compose의 MinIO로 검증, 운영은 `S3_ENDPOINT`만 비우면 AWS S3
- 검증을 `UploadValidator`로 분리: **매직 바이트로 실제 타입 판별**(JPEG/PNG/GIF/WebP),
  확장자도 판별된 타입에서만 유도. 클라이언트 신고값은 아예 사용하지 않음.
- MinIO 기동 시 버킷 자동 생성 + GetObject만 public-read 정책 (조회는 앱 서버 미경유).
- 단위 테스트 4개 추가 (위장 HTML 거부 포함) — 전체 81/81 통과.

**배치 주기 조정** — 기차 시간표 staleness 20h → 7일, 버스 6일 → 14일.
시간표는 분기 단위 개정인데 20h 기준은 "전 노선 매일 갱신"이라 KTX 쌍 수(272)가
일 예산(200)을 초과 → 일부 노선이 영구 미갱신되는 구조였음. 주기를 개정 빈도에
맞추면 같은 예산이 신규 노선 워밍에 쓰인다.

## 7. "안 뜨는 지도" 진단 — 콜드 지역 162초 → 새벽 선워밍 배치

**진단 (실측)**: 울산 콜드 조회 = 22개 구역 수집 + 1,002개 장소 Naver 블로그 채점 = **162초, HTTP 200**.
서버는 정상적으로 응답하지만 프론트 타임아웃(suggest 180s, 지역검색·코스생성 120s)과
아슬아슬 — 장소가 더 많은 지역이나 429 백오프가 겹치는 날은 타임아웃 = 지도가 빈 채로 뜸.
근본 원인은 429 자체가 아니라 **429를 피하려고 초당 제한에 맞춰 천천히 채점하는 설계**
(정확히는 그 시간 동안 사용자를 기다리게 하는 것).

**해결: RegionPlacesWarmupJob (매일 05:00)**
- RegionSeedData 49개 지역의 Redis 캐시(`places:...:ALL:지역명`)를 순회:
  키 없음(콜드) 우선 → 남은 TTL 2일 미만(만료 임박) 순으로 **하루 10개 지역만** 워밍.
- 쿼터 예산: 콜드 1지역 ≈ 최대 ~1,000 블로그 호출 → 10지역 ≈ 10,000 = Naver 일 쿼터(25,000)의
  40% 이하. 낮 시간 실사용 몫을 남긴다. 저장 TTL 7일 기준 5일 주기로 전 지역 순환.
- region 진입·GPS 진입 모두 지역 표준 중심좌표로 스냅되므로(7/1 devlog) 중심좌표 하나만
  데우면 모든 진입 경로가 캐시 적중.
- 교통 배치(04:00)와 시간대 분리. 순차 실행 — Naver 속도 제한을 실사용과 경합하지 않도록.
- 단위 테스트 2개: 일일 예산 상한 / 콜드→임박 우선순위·신선 지역 건너뜀.

**부수 수정**: 업로드 검증 예외를 `IllegalArgumentException`(500으로 새던 것) →
`GadangException.badRequest`(400 + 메시지)로 컨벤션 통일. E2E로 확인:
위장 HTML 400 거부, 정상 PNG 업로드 → MinIO 공개 URL GET 200 (image/png).

**전체 테스트 83/83 통과.** (77 → 81(이미지 검증 4) → 83(워밍 배치 2))

## 8. "알려진 한계" 2건을 그냥 고침 — JWT 매 요청 DB 조회 · LocalTime 자정 래핑

### ① JWT 필터 매 요청 DB 조회 → 60초 캐시 + 즉시 evict

- 문제: 필터가 서명 검증 후에도 요청마다 `userMapper.findById` — 인증 요청 수 = users SELECT 수,
  DB 장애가 전체 401로 전파. stateless JWT의 이점을 DB 왕복으로 되사는 구조.
- 해결: `AuthUserCache` (Caffeine, 60s TTL, 탈퇴 유저 empty도 캐시).
  탈퇴·권한 변경(AdminService)에서 `evict()` 호출 → TTL을 기다리지 않고 즉시 반영.
- 한계 문서화: evict는 인스턴스 로컬 — 수평 확장 시 타 인스턴스는 최대 60초 지연.
  필요해지면 Redis pub/sub 브로드캐스트. (refresh 회수 + access 30분이 이미 상한을 제공)

### ② LocalTime 자정 래핑 가드

- 문제: `LocalTime`은 24h 모듈러 — 23:50+40분=00:30이 되고 `00:30.isAfter(23:00)=false`라
  귀가 초과 검증이 뚫리고, 여유 시간이 +22시간으로 계산되는 "조용히 틀리는" 버그.
- 해결: `advance()`(누적 경로 17곳 — 래핑 시 badRequest "자정을 넘어…")와
  `wrapsPastMidnight()`(후보 선택 경로 6곳 — 래핑 후보는 skip/null) 이원화.
  일정 확정 경로는 명시적 에러, 탐색 경로는 해당 후보만 배제.
- 회귀 테스트: 23시 출발 서울→강릉(이동 수백 분) → 예전엔 검증 통과(버그), 이제 400 + "자정" 메시지.

테스트 83 → 87 (자정 회귀 1 + AuthUserCache 3). 실기동: 로그인·인증 요청 200 확인.
