# 2026-07-11 개발 로그

## 오늘 한 것 요약
- **장소 캐시 내구성 강화**: L2 Redis TTL 7일 → 30일 (만료로 인한 콜드 미스 감소)
- **선워밍 배치 예산 상향**: 10 → 18지역/일(~72%). 평소엔 TTL이 길어 만료 임박분만 소수 갱신,
  Redis 유실 시 전체 49지역을 ~3일에 재워밍하기 위한 상한 여유
- **네이버 일일 쿼터 게이트 신규**: 배치·온디맨드가 하나의 일일 예산(24,000)을 공유.
  쿼터 소진 시 외부 호출을 막아 검색 API 먹통을 원천 차단

---

## 배경 — 왜 필요했나
장소 후보 캐시는 `L1 Caffeine(휘발) → L2 Redis(TTL) → L3 네이버 블로그(쿼터 25,000/일)` 구조다.
Redis에 영속 볼륨이 없어(휘발), 낮 시간에 Redis가 죽거나 대량 만료되면 사용자 콜드 조회가
여러 지역으로 몰려 L3 외부 검색 API로 폭주 → 일 쿼터를 초과할 위험이 있었다.
(`@Cacheable(sync=true)`가 같은 키 동시요청은 1회로 묶어주지만, 서로 다른 지역 다발 미스는 못 막는다.)

## 설계 결정
1. **TTL 30일 + 배치 선워밍** — 만료 미스 자체를 줄이고, 만료 임박(3일 미만) 지역을 배치가 미리 갱신.
   TTL을 길게 잡아도 배치가 만료 전 갱신하므로 데이터 신선도는 유지된다.
2. **쿼터 게이트(NaverQuotaGate)** — "먹통"의 진짜 방어선. 모든 실제 블로그 호출(`fetchBlogCount`)이
   진입 전 게이트에서 1건씩 acquire. 예산 소진 시 false → 외부 호출 없이 0점 degrade.
   배치와 온디맨드가 같은 게이트를 공유하므로 둘을 합쳐도 일 예산을 넘을 수 없다.
   - 캐시 히트는 `fetchBlogCount`를 안 타므로 예산을 소비하지 않는다.
   - degrade로 빈 결과가 된 콜드 지역은 캐시에 저장되지 않아(skip-empty) 예산 리셋 후 자동 복구.
3. **왜 DB 백스톱이 아니라 게이트인가** — DB는 외부 호출 "횟수를 줄이는" 방법이고, 게이트는
   호출을 "쿼터 아래로 하드캡" 하는 더 근본적인 방법이다. 이 데이터는 우리가 소유한 원본이 아니라
   외부 API의 캐시라, DB 계층을 추가하기보다 소스(외부 API)를 게이트로 보호하는 쪽이 단순하고 확실하다.

## 면접 포인트
"캐시 붙였다"가 아니라 **내 설계의 실패 모드(Redis 유실 → 외부 API 콜드 스톰)를 스스로 진단하고
게이트로 하드캡** 한 서사. DB vs 게이트 트레이드오프(호출 횟수 감소 vs 하드캡)를 설명할 수 있다.
